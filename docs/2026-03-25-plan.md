## Priority 1: #96 — Model-aware chunk sizing across all ingest paths

**Why first:** Silent quality degradation on every query. Both reviewers independently identified this as the top priority.

**The real problem (verified in code):**
- `github.ts:253` — Issues/PRs emitted as single chunks (`totalChunks: 1`) with full `title + body`. No splitting.
- `slack.ts:151` — Message groups emitted as single chunks. No size limit.
- `discord.ts:165` — Same pattern.
- `openai.ts:66` — Silent character-based truncation (`maxInputChars` default 4000). No warning logged.
- `repo.ts:246` — Already chunks code at 512 chars. Not affected.
- `website.ts:51` — Already uses `chunkMarkdown()`. Not affected.

**Scope:**
1. Add model-aware chunk splitting to GitHub, Slack, and Discord adapters (or a shared post-processing step)
2. Replace character-based truncation with token-aware limits
3. Log warnings when truncation occurs (quick win, do first)
4. Fold #95 (embedding profiles) into this — one "profile" object per model with chunk size, token limit, dimensions

**Estimated complexity:** Medium. Adapter changes are mechanical; the design question is where splitting lives (per-adapter vs shared chunker).

## Priority 2: #94 — CID-based collection resolution (design first)

**Why second:** This is THE decentralized differentiator. Both reviewers agree on high impact but flag prerequisites.

**Prerequisites (do before or alongside):**
- #92 — Pull/import from FOC by CID (package-level bootstrap, not just server)
- #80 — Shared API contract (prevents ad-hoc endpoints in server)

**UX flow (from Codex, endorsed by Cursor):**
1. User opens hosted UI or pastes CID into "Open collection" input
2. Server classifies handle: head CID, revision CID, or segment CID
3. Fetch minimal metadata first, show loading state
4. Hydrate segments into cache-backed mounted collection
5. Switch to normal search/trace mode once queryable
6. Stable share URL keyed by CID, not local name

**Architectural gotchas (verified):**
- Server is hardcoded to `storage: "local"` in `apps/web/server/index.ts:75`
- `FocStorageBackend` requires a private key (`factory.ts`) — read-only public CID fetching needs a new composition
- `mountCollection()` exists in `packages/search/src/mount.ts` but server hand-rolls its own mounting
- Cache key strategy: head CID (mutable) vs revision CID (immutable) — prefer revision
- In-memory index will hit limits on large hosted collections — need eviction strategy
- Gateway/IPFS availability is flaky — need retry + partial-hydration states

**Recommended approach:** Thin vertical slice first (hardcoded happy path: known CID → gateway fetch → mount → query), then harden.

## Priority 3: Trace quality improvements

**New finding from Codex (verified):**
The trace docstring describes step 4 as "fall back to semantic search for unconnected source types" but the implementation only does:
1. Embed query → vector search for seeds
2. Follow edges from seeds
3. Group by source type

There is no second-pass semantic expansion for source types that got zero results from edge traversal. This means if edges only connect code→issues, doc-pages that are semantically relevant but unconnected by edges won't appear.

**Not "broken"** — the seed search does find results across source types. But the gap is real: if 3 doc-page seeds get found but all edge hops go to code/issues, doc-pages get underrepresented in the final grouped output.

**Scope:** Add a post-edge-traversal pass that checks which source types are missing from results and fills them with semantic matches. Small, high-impact change.
